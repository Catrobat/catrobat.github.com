Projects

Title: Gesture Control for Drones
Brief explanation:

Pocket Code will be extended by bricks to control Drones like the Parrot AR Drone Quadcopter. At GSoC' 15 a new brick which enables users to control the drone with hand gestures should be implemented. Therefore, as a first step, the recognition of these gestures through the front camera of the smartphone should be implemented. Afterwards Drone-commands to these gestures should be added, such as moving the hand up indicates the Drone to go up.

Expected results:

There will be small tasks split up by the mentor, so that this new feature will get part of the project step by step. In the end there should be a brick which recognizes gestures from the devices camera and reflect these to commands executed by the Drone. This brick should get part of the rest of the Drone project and be included into the app.

Mentor: Markus Hobisch
Title: Musicdroid Drumset-Synthesizer
Brief explanation:

In this task the student will create a simple drumset in form of a very small synthesizer. Therefore a grid of buttons will be created which represents beats of a rhythm. The lines of the grid represent different drumset-instruments and the columns are the beats. If one cell is set, the respective instrument gets played at the given beat. There should be a fragment for settings like the pace. This task also includes the creation of the sounds.

Expected results:

As part of Musicdroid this interface and drumset-synthesizer should be implemented and included in Musicdroid. Users can play the drums easily within their projects created with Musicdroid and combine it with other features of this app.

Mentor: Elisabeth Heschl or Andrej Mueller
Title: Musicdroid Ocarina Instrument
Brief explanation:

In this task the student will create a ocarina, which is a flute-like instrument. There should be buttons which represent the wholes of the instrument and the user should blow into the microphone to create a sound. The already implemented notesheet-view of the piano should also be included. The task includes the creation of the sound.

Expected results:

An user should be able to play the ocarina within Musicdroid. Therefore the described User interface and implementation has to be created and included into Musicdroid, to make it easy for everyone to play this instrument through musicdroid.

Mentor: Elisabeth Heschl or Andrej Mueller
Title: Object Eraser Tool for Paintroid
Brief explanation:

Implement a tool for our app Pocket Paint which let the user specify an area on the screen which will be erased from the image, similar to the 'Retouch' app on Google. That actually means that a selected area of an object within Pocket Paint should be replaced by its background. Therefore the infrastructure of Pocket Paint has to be considered.

Expected results:

At the end of GSoC this tool should be ready to be included into the app, so that an user just may select an area on his display and may erase it, to create even more complex graphics.

Mentor: Roland Urbano
Title: Lego NXT Sensor Handling
Brief explanation:

Sensors in the formula editor are currently hard coded. This means that there is a fixed assignment of a Sensors token (which is written to code.xml) and an R.string value which is shown in the user interface. But in the case of the LegoNXT robot it should be possible to change the assignment R.string value to a Sensors token at run-time. The lego robot has 4 slots where different sensors can be connected. It is possible to assign a sensor type (touch, light, ultrasonic, sound, etc.) to a sensor slot in the catroid settings. Because it is really troublesome to introduce all kind of sensors as tokens and save the them in the code.xml (e.g. nxt_touch_sensor, nxt_light_sensor), the sensor port (e.g. nxt_sensor_1, nxt_sensor_2) is used as token and get saved to the code.xml. So only the mapping of the name to the slot is saved and only this mapping must be changed when the assignment changes.

Expected results:

The sensor handling in Catroid is currently not very dynamic and it is quite complex to add new Sensors.With a dynamic sensor assignment, it should be possible to change the assignment of Sensor token (Sensors enum) to R.string values at run-time. All views which uses this R.strings should recognize this change and update properly. the context of introducing the dynamic sensor assignment it will be neccessary to refactor some parts of the sensor handling in the formula editor. So take this chance and refactor the sensor handling and abstract it from the formula editor. This refactoring should make it easy to add and manage sensor types afterwards. Also accessing sensor values should be simplified. Additionally it should be possible to show or hide sensor types based on the catroid settings. E.g. if lego nxt bricks are disabled also the nxt sensors should be hidden.

Mentor: Martin Burtscher
Title: Catrobat Infrastructure Automation
Brief explanation:

Currently the Catrobat project uses over 15 services and servers internally as well as facing to the public. Due to the complexity and more or less strict team-boundaries every sub-project within Catrobat built their own services on top of the provided servers. This resulted in a very diverse and isolated infrastructure architecture. Therefore a modern framework like Chef and Puppet should make it possible to create an automated workflow to describe the Catrobat services in a expressive code-like language. Each server and each service can be automatically updated and deployed from this collection of scripts and recipes.

Expected results:

Knowledge about how a server should be configured is accessible and stored in a version control system. Every team member can create pull requests to improve the configuration of a specific service. New servers are set up and initialised within seconds from the existing base recipe. Critical infrastructure like Catroweb servers, our URL-shortener, or JIRA can be recreated within minutes if the hardware fails. Every configuration and every change is comprehensible and open for inspection by the community. Security related issues are easily fixed with a single change to the repository which results in pushing out the new configuration to all servers and services part of the Catrobat infrastructure. Software updates and new technologies can be adapted more quickly, due to the clearly visible dependencies and architecture of the overall system. Teamwork and interaction between individual Catrobat teams is an important factor in running a large-scale community-driven project. Having a well-documented and easily changeable infrastructure is a key point in providing a good experience to our team members and to the public community of Pocketcode and Catrobat.

Mentor: Johannes Anton Rieder
