Title: Add a (time capsule like) history to Catroid
Brief explanation:

At first we want you to add simple undo / redo functionality into Catroid. This basically means that user mistakes in the scripts, sounds, costumes shall be un- or re-doable. Imagine it like a typical Undo/Redo Button you find in nearly all of your IDEs.

Mentor: Peter Treitler
Title: Scratch to Catrobat converter
Brief explanation:

Extend and update our converter for [http://scratch.mit.edu/ Scratch] Programs to Catrobat programs. This involves, but is not limited to, decoding the projects from Scratch, converting them to Catrobat programs, and automating the process. Our existing code base for this project, which already contains much of the needed functionality, can be found on https://github.com/Catrobat/ScratchToCatrobat Note that keyboard events are translated into new "key" objects, with all "When key XXX pressed" event bricks replaced by a small script in this new object that shows the corresponding image of the key on the screen at some appropriate place, and which has a "When tapped" script that sends a 'XXX-key pressed' broadcast message. Where the original key bricks were in the Scratch program, a "When 'XXX-key pressed' broadcast message is received" brick is automatically inserted. Kids can then change this manually to something more appropriate on a smartphone or tablet, e.g., by using the inclination sensors. This also works already but must be made more beautiful and intelligent. All Scratch blocks not yet implemented in Catrobat need to be translated into comments that show all relevant content and parameters.

Expected results:

You get a Scratch Program from [http://scratch.mit.edu/ here] and convert it into a Catrobat Program so that it can be played and edited in Pocket Code. There must be an interface in the administration section of our web interface to initiate this process. Such a web interface already exists in a rudimentary form. The project must be fully implemented, tested, and *INTEGRATED* into the ScratchToCatrobat repository on github.

Mentor: Bianca Teufl or Andreas Schmidhofer
Title: List support in Formula Editor
Brief explanation:

Lists should be implemented in the Formula Editor. Listelements can be any kind of Formula (NUMBERS, STRINGS, SENSORS, nested LISTS) Implementation of the following functionality is required: * Creation of a list ( listname, Scope: for all or this object only ). * Add an item to a list. * Delete first, last or all item of a list. * Insert an item at first, last or random position in a list. * Replace an item ( first, last or random ) with a new item in a list. * Return an item at first, last or random position in a list. * Length of a list. (Number of items in a list) * Does a list contain a certain value? (Boolean function) Optional functionality: * Displaying of lists on stage. (listname, listindex, interpretation of listitem, length of list) * Show list on stage brick. * Hide list on stage brick.

Expected results:

This task is split up from the mentor so that you will always have to implement one small task after the other. At the end of GSoC, the agreed-upon functionality must be fully implemented, tested, and *INTEGRATED* into the Pocket Code master branch.

Mentor: Anton Rieder or Stefan Jaindl
Title: Search for Images
Brief explanation:

It should be possible to add images (and potentially sounds) by mining CC-SA-BY media content (creative commons). This is content that can be used when the license and author are given attribution. Touchdevelop by implements a similar functionality (https://www.touchdevelop.com/ ). As a first step the content should be searchable and insertable only from local Pocket Code programs on the file system and the Pocket Code server. New xml entries are required for each media file. The entries should contain licenses including the original author, plus any other meta information, e.g., a link to the place where and when the files were found on the web or at some other place, and any other information, e.g., keywords or tags etc. Optionally, credits might be given in textual form (as another xml entry), both for single media files as well as for whole Catrobat programs. This project may be realized as a Web API. As a next step it would be nice to search on Scratch, Touch-develop, Google Image - but this is not a requirement, it is nice to have.

Expected results:

This task is split up from the mentor so that you will always have to implement one small task after the other. At the end of GSoC, the agreed-upon functionality must be fully implemented, tested, and *INTEGRATED* into the appropriate master branches.

Mentor: Marko Burazer or Theresa Egger
Title: Cloning Bricks
Brief explanation:

Cloning in Scratch 2.0 is a feature that allows a sprite to create a clone, or duplicate, of itself, while the project is running. This can be useful in tower defense games, for example, for a wave of objects. Clones of a sprite will be the same as the original, or parent sprite, but as a separate instance (see http://wiki.scratch.mit.edu/wiki/Cloning). 3 additional bricks are required - these bricks are "create a clone", "delete clone" and as script brick "when I start as clone", just like in Scratch.

Expected results:

Mentor: Stefan Simon
Title: Make Pocket Code Bricks feature-complete to Scratch
Brief explanation:

Pocket Code is missing the popular show/hide variable as well as pen bricks from Scratch. The aim of this project is to implement these missing bricks. Show and hide variable bricks are needed in Pocket Code programs. They are necessary to display, e.g., game statistics. See http://wiki.scratch.mit.edu/wiki/Show_Variable_()_(block) . The Show Variable () block is a Variables block and a Stack block. The block shows the specified variable's Stage monitor. In projects that use variables' Stage monitors to display things, the displays may have to hide and show at times; this can be done with the Hide Variable () brick. Pen blocks in Scratch 2.0 are used to draw something on the screen, much like in Turtle graphics. There are eleven bricks in this category. Pen bricks allow to take various actions: - Stamp the sprite - Put the pen down - Release the pen One example of a one sprite one script project is a simple drawing program. See http://wiki.scratch.mit.edu/wiki/Pen_Blocks

Expected results:

This task is split up from the mentor so that you will always have to implement one small task after the other. At the end of GSoC, the agreed-upon functionality must be fully implemented, tested, and *INTEGRATED* into the Pocket Code master branch.

Mentor: Wolfgang Slany
Title: Global User-Defined Bricks
Brief explanation:

Currently (in the version from a GSoC 2013 project) the user-defined bricks are always local to one object. The aim of the new GSoC project would be to extend this functionality to user-defined bricks that are global to a Catrobat program. Additionally, copy/cut/pasting should also be supported in all possible ways, e.g., to make a duplicate of a local or global user-created brick, or copy a local one and paste it into another object. Also, when copying objects, any local user-defined bricks should also be copied, and when copying scripts any user-defined bricks that were used in the scripts should also be included in the copy.

Expected results:

This task is split up from the mentor so that you will always have to implement one small task after the other. At the end of GSoC, the agreed-upon functionality must be fully implemented, tested, and *INTEGRATED* into the master branch.
